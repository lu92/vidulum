VID-161 — Onboarding: Bank Accounts

Why we ask

When a user transfers money between their own bank accounts (e.g. Nest → Pekao for monthly savings), it looks like a regular expense in their CashFlow. Without knowing that the destination IBAN belongs to the user, the system inflates "expenses" by these transfer amounts and makes the budget unreliable.

Without onboarding
22,500 PLN / month
Inflated by ~9,000 PLN of fake "expenses" (self-transfers)
→
With declared accounts
13,500 PLN / month
Real expenses only — transfers excluded from budget
Real-world impact: in our Nest Bank test data, 111 out of 402 transactions (50.7%) are self-transfers to user's own Pekao / mBank / ING accounts. A 2-minute onboarding eliminates this noise on day one, before the first CSV import.

Why ask upfront (not later via suggestions)?

  • Immediate value — first CSV import already shows correct budget, no "Inne" pollution
  • Educational moment — user understands why the system asks, increases buy-in
  • DRY for CashFlow creation — IBAN entered once, reusable for any future CashFlow
  • Deterministic detection — IBAN matching needs no heuristics, zero false positives

CashFlow details

Critical UX contract: when user picks an account from the picker, the UI pre-fills the CashFlow creation payload's bankAccount with that account's data (iban, currency, bankName). Nothing else is needed — backend's Kafka listener resolves the link automatically: if the IBAN is already in the profile, it sets linkedCashFlowId on the existing entry; if not, it claims as a new entry with source=CASHFLOW.

4. Backend integration — REST APIs used

Authentication

All endpoints below require a valid JWT in the Authorization: Bearer <token> header. The token is obtained from POST /api/v1/auth/register (during the auth step that precedes onboarding). The user's identity is derived from the token — the UI does not need to pass userId explicitly.

Endpoints used by the onboarding + picker flow

Method Endpoint Purpose / when called
POST /api/v1/auth/register Step 1 of onboarding — user signs up. Backend creates the User and an empty UserFinancialProfile document atomically. UI captures access_token for subsequent calls.
POST /api/v1/user/owned-accounts/bulk Onboarding "Save and continue" button. Submits all accounts in one call. Backend validates every IBAN (all-or-nothing) and on success emits one OwnedBankAccountAddedEvent per account on the user_financial_profile Kafka topic.

Source on backend side is ONBOARDING.
GET /api/v1/user/owned-accounts/available-for-cashflow CashFlow picker dropdown. Returns accounts where status=ACTIVE AND linkedCashFlowId IS NULL (i.e. owned but not yet attached to any CashFlow). Called when the "Create CashFlow" screen opens AND after the quick-add modal closes successfully.
POST /api/v1/user/owned-accounts Quick-add modal "Save and use" button. Adds a single account. Source is MANUAL. Returns the new OwnedAccountJson including the assigned IBAN (normalized). UI uses this IBAN to refresh the picker and auto-select.
POST /cash-flow/with-history CashFlow "Create" button. Standard payload — bankAccount with the IBAN that the user picked (or quick-added). Backend creates the CashFlow and emits a CashFlowWithHistoryCreatedEvent on the cash_flow topic; the UserFinancialProfileCashFlowListener auto-links the IBAN to an existing OwnedBankAccount in the profile, or claims it as a new entry if missing (eventually consistent — typical lag < 500ms).
GET /api/v1/user/owned-accounts Optional — used by "My Bank Accounts" settings page (Phase 3). Returns the full list with all statuses and source markers. Not directly part of the onboarding flow but useful for confirming the saved data.

Error handling — what the UI must surface

HTTP status ErrorCode When / UI handling
400 VALIDATION_ERROR Missing required field (iban / currency / bankName / label is blank). Show inline field error from fieldErrors[].
400 INVALID_BANK_ACCOUNT IBAN checksum invalid. Show inline "Invalid IBAN" hint on the offending row.
409 OWNED_ACCOUNT_ALREADY_EXISTS IBAN duplicate within the bulk batch OR collision with an account already in the user's profile. Show "You've already added this account" and let user remove duplicate.
422 OWNED_ACCOUNT_CASHFLOW_LINKED User tried to delete an account currently linked to an active CashFlow. Show "Delete the CashFlow first" with a link to that CashFlow.

5. End-to-end flow (UI events ↔ backend events)

1
User registers
POST /api/v1/auth/register → backend creates User + empty UserFinancialProfile. UI receives access_token + user_id. Stores token, navigates to onboarding.
2
User fills the onboarding form
UI maintains local state of the account rows. On IBAN blur: client-side IBAN validation + bank-name auto-detect (table lookup on bank code). User can add/remove rows freely.
3
User clicks "Save and continue"
UI calls POST /api/v1/user/owned-accounts/bulk with the array of accounts. Disable the button while in-flight. On 201: navigate to the next onboarding step (or dashboard). On 4xx: show the relevant error inline (see error table above).
4
User skips onboarding
No API call. Profile stays empty. User can add accounts later from Settings or via the inline quick-add inside CashFlow creation. The motivation copy on the onboarding screen should make the trade-off clear ("you can add later, but your first imports will be less accurate until you do").
5
Later: user creates a CashFlow
UI calls GET /api/v1/user/owned-accounts/available-for-cashflow to populate the picker dropdown. User picks one. UI pre-fills the bankAccount payload from the picked account's data.
6
User clicks "+ Add new bank account" in the picker
Modal opens. UI keeps the CashFlow form state intact. On modal submit: POST /api/v1/user/owned-accounts (source=MANUAL). On 201: close modal, re-call GET /available-for-cashflow, auto-select the newly added entry, keep CashFlow form visible.
7
User clicks "Create CashFlow"
UI calls POST /cash-flow/with-history with the full bankAccount payload. Returns CashFlow ID. UI navigates to the new CashFlow page. Meanwhile (async, ~500ms): the listener on cash_flow topic looks up the IBAN in the user's profile and either sets linkedCashFlowId on the existing entry or claims it as a new entry. User does not see this — it's purely backend bookkeeping for future self-transfer detection.

6. Out of scope (Phase 2+)

  • Post-import suggestions — system analyses CSV counterparty accounts and suggests "this IBAN you transferred to looks like yours". User accepts/dismisses. Documented in VID-161 Phase 2 TODO.
  • CashFlow delete lifecycle — when a CashFlow is removed, linkedCashFlowId should be cleared on the OwnedBankAccount. Tracked as separate TODO.
  • Closed accounts — distinguishing "account closed at bank but historical transactions still reference it" from "active account". Not part of onboarding.
  • Multi-currency UX — current mockup defaults to PLN. Multi-currency dropdown shown but no smart inference. Acceptable for Phase 1.
